# Существующие MCP-серверы в области зависимостей и лицензий

Обзор серверов, решающих смежные задачи: безопасность зависимостей, управление пакетами, лицензии и dependency intelligence.

---

## 1. BoostSecurity MCP (`boost-mcp`)

**Репозиторий:** [boost-community/boost-mcp](https://github.com/boost-community/boost-mcp)  
**Платформа:** [BoostSecurity](https://boostsecurity.io/)

### Назначение

Защита агентных воркфлоу от небезопасных зависимостей: проверка каждой пакета, которую ИИ-агент предлагает добавить в проект.

### Функциональность

| Аспект | Описание |
|--------|----------|
| **Инструменты** | `validate_package` — проверка пакета на безопасность и получение рекомендации (в т.ч. альтернативы). |
| **Риски** | Уязвимости, EOL, malware, typosquatting, несуществующие пакеты (галлюцинации агента). |
| **Экосистемы** | PyPI, Go Modules, npm, Maven, NuGet. |

### Сильные стороны

- Мульти-экосистема в одном инструменте.
- Чёткая роль: «сторож» перед добавлением зависимости.
- Удалённый HTTP MCP (https://mcp.boostsecurity.io/mcp), не требует локальной установки сканеров.

### Ограничения

- Нет полного аудита всего проекта — только проверка одного пакета.
- Лицензии не в фокусе.
- Нет SBOM, дерева зависимостей, политик.

## 2. NPM Plus (`js-package-manager-mcp`)

**Репозиторий:** [shacharsol/js-package-manager-mcp](https://github.com/shacharsol/js-package-manager-mcp)  
**Хостинг:** https://api.npmplus.dev/mcp

### Назначение

Интеллектуальный менеджер npm-пакетов для ИИ: поиск, установка, обновление, аудит и лицензии в контексте реального проекта.

### Функциональность

| Инструмент | Описание |
|------------|----------|
| `search_packages` | Поиск в npm registry с релевантностью. |
| `package_info` | Метаданные пакета. |
| `install_packages` / `update_packages` / `remove_packages` | Управление зависимостями (npm/yarn/pnpm). |
| `audit_dependencies` | Сканирование уязвимостей по проекту. |
| `check_vulnerability` | Проверка конкретного пакета на уязвимости. |
| `check_license` | Лицензия одного пакета. |
| `list_licenses` | Список лицензий по проекту. |
| `dependency_tree` | Визуализация дерева зависимостей. |
| `analyze_dependencies` | Циклические зависимости, сироты. |
| `check_outdated` | Устаревшие пакеты. |
| `check_bundle_size` | Анализ размера бандла. |

### Сильные стороны

- Богатый набор инструментов для JS/TS; удобен как «умный npm CLI» для ИИ.
- Есть и уязвимости, и лицензии (`list_licenses`, `check_license`).
- Поддержка относительных путей (`.`), кэширование, retry при установке.

### Ограничения

- Только JavaScript/TypeScript (npm/yarn/pnpm).
- Лицензии — список и проверка, без политик и глубокого анализа (copyleft, совместимость).
- Нет SBOM, нет мульти-языкового сводного отчёта.

## 3. Maven Tools MCP (`maven-tools-mcp`)

**Репозиторий:** [arvindand/maven-tools-mcp](https://github.com/arvindand/maven-tools-mcp)  
**Доставка:** Docker-образ `arvindand/maven-tools-mcp`

### Назначение

Dependency intelligence для JVM: актуальные версии, совместимость, миграции (например, Spring Boot 2 → 3), CVE и рекомендации по обновлению.

### Функциональность

| Аспект | Описание |
|--------|----------|
| **Источники** | Maven Central, анализ pom.xml / Gradle. |
| **Возможности** | Сравнение текущей и последней версий, стабильность (stable/RC/beta), breaking changes, совместимые версии зависимостей. |
| **Интеграции** | Context7 — документация и гайды по миграции. |
| **Безопасность** | Проверка CVE, рекомендации по безопасным версиям. |

### Сильные стороны

- Глубокое понимание JVM-экосистемы и миграций.
- Подходит для код-ревью апгрейд-PR (Renovate/Dependabot) и для агентов, которые сами обновляют зависимости и правят код.

### Ограничения

- Только JVM (Maven, Gradle, SBT, Mill).
- Лицензии упоминаются (например, «2 GPL dependencies») в отчёте о здоровье, но не являются основным фокусом.
- Нет кросс-языкового анализа, нет SBOM в явном виде в описании.

## 4. VulniCheck MCP (`vulnicheck`)

**Репозиторий:** [andrasfe/vulnicheck](https://github.com/andrasfe/vulnicheck)  
**Доставка:** Docker `andrasfe/vulnicheck`

### Назначение

Сканер безопасности для Python-проектов и репозиториев: уязвимости, секреты, Dockerfile, опционально — ИИ-оценка рисков.

### Функциональность

| Инструмент | Описание |
|------------|----------|
| `check_package_vulnerabilities` | Уязвимости конкретного Python-пакета. |
| `scan_dependencies` | Скан requirements.txt, pyproject.toml, setup.py и lock-файлов. |
| `scan_installed_packages` | Скан установленных в окружении пакетов. |
| `get_cve_details` | Детали CVE. |
| `scan_for_secrets` | Поиск секретов и учётных данных в коде. |
| `scan_dockerfile` | Анализ Dockerfile на уязвимые зависимости. |
| `scan_github_repo` | Полное сканирование GitHub-репозитория. |
| `assess_operation_safety` | ИИ-оценка риска операции. |
| `validate_mcp_security` | Проверка конфигурации MCP на безопасность. |
| `comprehensive_security_check` | Комплексная интерактивная проверка. |

**Базы уязвимостей:** OSV, NVD, GitHub Advisory, CIRCL, Safety DB.

### Сильные стороны

- Несколько баз уязвимостей, поддержка lock-файлов, CVE-детали, CVSS.
- Дополнительно: секреты, Dockerfile, GitHub repo scan, опциональная аутентификация (Google OAuth).
- Работает по HTTP (streamable), без обязательного SSE.

### Ограничения

- Только Python (и контейнеры/репо вокруг него).
- Лицензии не в фокусе.
- Нет мульти-языкового проекта, нет SBOM и политик лицензий.

## 5. Snyk MCP-расширение (`snyk-ls`)

**Контекст:** Snyk Language Server / расширение с MCP-интеграцией.  
**Документация:** [snyk-ls MCP extension](https://github.com/snyk/snyk-ls/blob/main/mcp_extension/README.md)

### Назначение

Встраивание Snyk SCA (уязвимости, лицензии, при необходимости SBOM) в MCP-клиенты, чтобы ИИ мог использовать данные Snyk в диалоге.

### Функциональность

- Типичный функционал Snyk: сканирование зависимостей, уязвимости, рекомендации по апгрейду.
- Лицензии и политики — часть платформы Snyk, доступ через общий SCA-стек.
- Интеграция через MCP даёт агентам доступ к результатам сканирования.

### Сильные стороны

- Зрелая SCA-платформа (много экосистем, лицензии, политики, CI/CD).
- Единый дашборд и политики в Snyk; MCP расширяет использование в ИИ-сценариях.

### Ограничения

- Привязка к SaaS Snyk (учётная запись, API keys).
- Нет полностью локального/офлайн сценария без Snyk.
- Документация MCP-расширения минимальна; точный список MCP tools нужно уточнять по коду.

## Выводы и ниша для «Инспектора зависимостей и лицензий»

- Существующие MCP закрывают **отдельные куски**:
  - **Безопасность пакетов:** BoostSecurity (превентивно), Snyk, VulniCheck.
  - **Управление + базовые лицензии:** NPM Plus (только JS).
  - **Dependency intelligence:** Maven Tools (только JVM).

- **Пробелы**, которые заполняет целевой MCP «Инспектор зависимостей и лицензий»:
  1. **Кросс-языковой анализ** в одном сервере: JS/TS, Python, опционально JVM — единый граф и отчёт.
  2. **Глубокий лицензионный анализ**: политики (allow/deny/comment), риски copyleft, совместимость лицензий, отчёты для юристов.
  3. **SBOM** и структурированные отчёты: генерация SPDX/CycloneDX и человекочитаемых отчётов для безопасников и compliance.
  4. **Локальная работа**: анализ по файлам проекта без обязательной привязки к SaaS; при необходимости — интеграция с OSV, npm audit, pip-audit и т.д.
  5. **Единая точка входа** для сценариев «аудит репо», «отчёт для юриста», «проверка политики», «подбор замен» в диалоге с ИИ.
