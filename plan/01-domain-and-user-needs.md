# Домен и потребности пользователей

## Проблема

Современные проекты сильно зависят от открытого кода: до 96% коммерческих кодовых баз содержат open-source компоненты, у 84% есть хотя бы одна известная уязвимость. При этом:

- **Зависимости разбросаны по экосистемам**: npm, PyPI, Maven, Go modules и т.д. Нет единого места, где можно увидеть полную картину по проекту.
- **Лицензии создают юридические риски**: GPL, AGPL, LGPL, SSPL и другие лицензии по-разному влияют на продукт.
  Разработчикам и юристам сложно быстро понять, какие зависимости допустимы по корпоративной политике.
- **Уязвимости требуют приоритизации**: не каждую CVE нужно чинить срочно; важно понимать, что в критическом пути, а что — в dev-зависимостях.
- **Мост от сырых данных к решениям строится вручную**: SCA-инструменты выдают списки; интерпретация, план действий и отчёты для стейкхолдеров — за человеком.

## Целевые пользователи

### Разработчики (в IDE / AI-редакторах)

- **Потребности**:
  - Перед релизом или PR — быстро проверить проект на уязвимости и лицензии.
  - Понять, какие зависимости устарели и какие риски несут.
  - Получить конкретные команды или патчи для устранения проблем, а не только сырые данные.
- **Pain points**:
  - Разные инструменты для разных языков (npm audit, pip-audit, Maven plugins и т.д.).
  - Сложно оценить, какая лицензия реально создаёт риск для продукта.
  - Мало интерактивных подсказок «что сделать дальше» в контексте чата с ИИ.

### Инженеры по безопасности (AppSec / SecOps)

- **Потребности**:
  - Быстрый аудит нового или унаследованного репозитория.
  - Проверка соблюдения корпоративной политики лицензий (например, запрет copyleft в ядре, но разрешение в тестовых инструментах).
  - Ответы на запросы вида: «Покажи все GPL-зависимости в продакшен-пути».
- **Pain points**:
  - Нужен единый отчёт по всему репо, а не по одному языку.
  - Не всегда есть прямой доступ к коду/CLI; при этом доступ к ИИ-инструментам (Cursor, Claude и т.д.) есть.

### Юристы и Compliance

- **Потребности**:
  - Короткие агрегированные отчёты по лицензиям с объяснением рисков и вариантов действий.
  - Понимание «что это значит юридически», а не только список имён лицензий.
- **Pain points**:
  - Сырые отчёты SCA сложны для интерпретации.
  - ИИ может выступать переводчиком из технических деталей в юридический контекст, если у него есть структурированные данные от MCP.

## Сценарии использования

1. **Проверка перед PR**: разработчик просит ИИ «проверь зависимости и лицензии в этом проекте» — MCP сканирует проект и возвращает сводку; ИИ формулирует выводы и рекомендации.
2. **Аудит нового репо**: безопасник подключает репозиторий и запрашивает «полный отчёт по уязвимостям и лицензиям» — MCP строит граф зависимостей, сканирует уязвимости и лицензии,
   проверяет политику; результат — приоритизированный отчёт.
3. **Отчёт для юриста**: пользователь запрашивает «отчёт для юриста по лицензиям» — MCP отдаёт структурированные данные; ИИ формирует краткий текст с рисками и вариантами (заменить, исключить, принять).
4. **Рекомендации по замене**: для проблемной зависимости (CVE или недопустимая лицензия) пользователь просит альтернативы — MCP ищет безопасные/допустимые аналоги
   и возвращает список с обоснованием.

## Pain points (сводка)

| Персона | Боль |
|--------|-----|
| Разработчик | Разбросанные инструменты по языкам; непонятно, какая лицензия создаёт риск; мало подсказок «что делать дальше». |
| Безопасник | Нужен единый отчёт по репо; проверка политики лицензий; не всегда есть доступ к CLI/коду. |
| Юрист/Compliance | Сырые отчёты сложны; нужны краткие выводы и варианты действий. |
| Общее | Мост от сырых результатов SCA к осмысленным рекомендациям строится вручную; MCP даёт шанс встроить анализ в диалог с ИИ и агентные сценарии. |
