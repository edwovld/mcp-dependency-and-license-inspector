# 07. Дальнейшие шаги и использование Composer 1.5 кредитов

## 1. Цели использования Composer 1.5

- Максимально эффективно потратить промо‑кредиты на «тяжёлые» и творческие задачи.
- Получить дополнительные материалы вокруг MCP‑сервера без изменения его детерминированного ядра.
- Подготовить документацию и материалы для демонстрации/защиты проекта.

## 2. Документация и объяснения для разных аудиторий

- Сгенерировать альтернативные версии `README.md`:
  - краткий питч (1–2 абзаца) для руководства,
  - подробное объяснение для разработчиков и AppSec.
- Подготовить высокоуровневое описание продукта на английском (для GitHub / внешних материалов).
- Сформировать FAQ по MCP‑серверу: 10–15 вопросов и ответов.

## 3. DEMO‑сценарии и кейсы использования

- Расширить `DEMO.md` новыми сценариями:
  - монорепозиторий с несколькими сервисами,
  - проект с жёсткой лицензионной политикой,
  - проект, где важна история миграции зависимостей.
- Попросить варианты «экстремальных» кейсов:
  - очень большой `requirements.txt`,
  - смешанные экосистемы (Python + Node),
  - конфликтующие политики лицензий.

## 4. Материалы для презентаций и выступлений

- Отшлифовать текст слайдов презентации (`presentation/dependency-inspector-mcp-slidev.md`) без разметки:
  - выровнять стиль и терминологию,
  - получить предложения по более чётким формулировкам.
- Подготовить черновик «скрипта докладчика»:
  - тезисы к каждому слайду,
  - возможные комментарии и переходы.
- Сгенерировать список вопросов от жюри/аудитории и примерные ответы:
  - архитектура и ограничения,
  - безопасность и политика лицензий,
  - сравнение с существующими SCA‑решениями.

## 5. Roadmap и развитие MCP‑сервера

- Сформировать варианты roadmap v2 и v3:
  - дополнительные экосистемы (Maven/Gradle и др.),
  - интеграции с CI/CD,
  - готовые отчёты для compliance/юристов.
- Проработать сценарии коммерциализации:
  - как MCP‑сервер может использоваться в компаниях,
  - какие метрики и отчёты важны для бизнеса.

## 6. Анализ конкурентов и позиционирование

- Попросить краткий обзор типичных возможностей SCA‑инструментов (Snyk, Dependabot, GitHub Advanced Security и др.).
- Сопоставить наш MCP‑сервер с такими инструментами:
  - где мы сильнее (интеграция в MCP, детерминизм, DEMO без платных API),
  - где слабее и что можно добавить в будущем.

## 7. Шаг «Composer 1.5: стратегическая сессия»

Использовать Composer 1.5 для одной большой стратегической сессии по проекту. Пример промпта, который можно вставить в Composer:

```markdown
Ты — ведущий архитектор и продуктовый менеджер MCP‑сервера «Инспектор зависимостей и лицензий».

У меня есть детерминированный MCP‑сервер для анализа зависимостей, лицензий, уязвимостей и проверки политики лицензий. Ядро написано на Python 3.11, использует MCP SDK (stdio), Pydantic, tomli, ruamel.yaml, httpx, OSV (или стабы в DEMO_MODE). В репозитории уже есть:

- планы `plan/01–06` с доменом, архитектурой и реализацией,
- рабочий сервер с tools: analyze_project_dependencies, scan_vulnerabilities, scan_licenses, check_policy_compliance, generate_sbom, suggest_dependency_replacements,
- DEMO‑сценарий, Docker‑контракт (serve/smoke), тесты и презентация на Slidev.

Задача: помочь мне спланировать следующие 2–3 итерации развития проекта и материалы вокруг него.

Сделай, пожалуйста:

1. Краткое резюме продукта (2–3 абзаца) на русском и английском, ориентированное на технического руководителя.
2. Предложи roadmap v2 и v3 (по 3–5 фич на каждую версию) с упором на:
   - новые экосистемы/языки,
   - интеграции с CI/CD,
   - дополнительные отчёты для безопасности и юристов.
3. Придумай 3–4 новых DEMO‑кейса, которые хорошо показывают ценность MCP‑сервера (опиши шаги и ожидаемые результаты).
4. Составь список из 15–20 возможных вопросов от жюри/аудитории на защите проекта и дай краткие, но содержательные ответы.
5. Предложи улучшения для текущей презентации: какие слайды добавить/объединить, какие метрики или диаграммы стоит показать.

Формат ответа:

- Разделы 1–5 с нумерацией.
- В каждом разделе — списки и короткие, но информативные формулировки, без воды.
```

Этот промпт можно адаптировать под актуальное состояние репозитория, но его структура должна оставаться примерно такой, чтобы извлечь максимум пользы из Composer 1.5 кредитов.
