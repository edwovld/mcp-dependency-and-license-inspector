# План реализации: детерминированный MCP-сервер «Инспектор зависимостей и лицензий»

Пошаговый план с учётом требований: тестируемость без LLM, демо без платных сервисов/ключей, ограничения по ресурсам и безопасность репозитория.

---

## 1. Цели и ограничения (из промпта)

| Требование | Описание |
|------------|----------|
| **Детерминированность** | Результаты инструментов MCP однозначно определяются входными данными (манифесты, конфиг). Без вызовов LLM в ядре. |
| **Тестируемость без LLM** | Все сценарии (unit, integration) проходят по фикстурам и мокам; LLM не требуется. |
| **Базовый сценарий DEMO.md** | Работает без внешних платных сервисов и API-ключей (OSV — бесплатный; при отсутствии сети — заглушки/демо-данные). |
| **Docker image** | ≤ 500 МБ. |
| **Ресурсы** | CPU: 2.0, RAM: 2048 MB. |
| **Безопасность репо** | Не хранить секреты, токены, ключи, пароли; не коммитить `.env` с реальными значениями. |
| **Конфигурация** | `.env.example` без секретов; заглушки/демо-конфиги для локального и CI-запуска. |

---

## 2. Обновления относительно планов 01–05

- **Ядро сервера** — только детерминированная логика: парсинг манифестов, классификация лицензий, сравнение с политикой, сборка SBOM. Запросы к внешним API (OSV, npm registry, PyPI) — опциональны и изолированы; при недоступности или в демо-режиме используются стабы.
- **Демо-режим** — переменная окружения `DEMO_MODE=true` или отсутствие сети: инструменты возвращают предсказуемые данные из фикстур/стабов, без платных ключей.
- **Тесты** — полностью на фикстурах (манифесты, ожидаемые JSON/SBOM); моки для HTTP (OSV, registry); без вызовов реальных API и без LLM.
- **Документация** — в конце реализации создаётся `DEMO.md` с пошаговым сценарием, который можно выполнить локально без ключей и платных сервисов.

## 3. Архитектура MCP: принципы и шаги

Вся архитектура сводится к одной идее: каждая функция, помеченная декоратором `@mcp.tool`, становится доступной агенту как инструмент. SDK упрощает регистрацию инструментов, но не решает архитектурные задачи. При создании сервера учитываются принципы:

| Принцип | Реализация в проекте |
|--------|----------------------|
| **Сигнатура — контракт** | Типы и структура возвращаемого значения стабильны; при ошибках бизнес-логики возвращается явный объект с полем `error` (и при необходимости `success: false`), а не текст «вперемешку» с данными. Ошибки валидации (например, несуществующий путь) передаются через исключения → MCP возвращает ошибку клиенту. |
| **Описание инструмента — часть интерфейса** | Описание инструмента в декораторе и **описание каждого параметра** (через `Annotated[тип, "описание"]` или `Field(description=...)`) влияют на то, как агент вызывает инструмент. Агент читает именно эти описания. |
| **Ошибки явно** | Исключения и ошибки возвращаются явно: либо через raise (протокол MCP вернёт error клиенту), либо через структурированный ответ с полем `error`; не скрываются внутри успешного текста. |

### Шаг 1. Инициализация

В данном проекте используется **Python + FastMCP** (не TypeScript). Структура: `src/mcp_dependency_inspector/` (server.py, core/, models/, parsers/), конфигурация и зависимости — см. `plan/04-architecture-and-api.md` и `project-structure.txt`. Для серверов на TypeScript используется `npx @modelcontextprotocol/create-server dependency-inspector-server` (index.ts, tsconfig, Zod).

### Шаг 2. Tools

У каждого tool четыре составляющих: **имя** (в MCP), **описание** на естественном языке, **схема параметров** (типы + описания параметров), **обработчик** (async или sync-функция). Критически важно: у каждого параметра должно быть описание — в Python через `Annotated[тип, "описание"]` или `Field(description="...")`; агент читает именно эти описания. Имена инструментов в API согласованы с планом: `analyze_project_dependencies`, `scan_vulnerabilities`, `scan_licenses`, `check_policy_compliance`, `generate_sbom`, `suggest_dependency_replacements`.

**Описание инструмента — промпт для модели, а не комментарий для человека.** Модель по name и description выбирает, какой tool вызвать. Расплывчатое описание («работает с данными») приводит к тому, что нужный инструмент не будет вызван; конкретное описание (что возвращает, какие аргументы принимает, что происходит в краевых случаях) даёт модели достаточно информации для выбора.

### Шаг 3. Resources

Текущее состояние данных в JSON — чтение без побочных эффектов. Реализованы: `report://latest`, `sbom://latest/spdx`, `sbom://latest/cyclonedx` (см. server.py).

### Шаг 4. Подключение

В конфиге MCP-клиента (`.mcp.json` или `mcp.json`) добавляется сервер: имя и команда запуска. Пример — в `DEMO.md`. После перезапуска IDE агент видит tools.

### Шаг 5. Отладка

`npx @modelcontextprotocol/inspector` — визуальный клиент, показывающий то, что видит агент: список tools, параметры, описания. Можно вызвать любой tool вручную и проверить контракт ответа.

### Транспорт и протокол MCP

Сервер соответствует требованиям MCP по транспорту и протоколу:

- **Два варианта транспорта:** **stdio** — для локальных серверов (хост запускает процесс, обмен через stdin/stdout); **Streamable HTTP** — для удалённых (HTTP + стриминг через Server-Sent Events, облачные деплои и Docker). Streamable HTTP заменил устаревший SSE-транспорт.
- **Протокол:** под капотом — **JSON-RPC 2.0**; каждый вызов инструмента — пара запрос/ответ. Реализация — через MCP SDK (mcp, fastmcp); код сервера не пишет JSON-RPC вручную.
- **Типизация:** аргументы инструментов описываются через **JSON Schema**; SDK формирует схему из сигнатур и аннотаций (`Annotated[тип, "описание"]`), модель получает точное описание аргументов.

### Шаг 2. Структура проекта и зависимости

- [x] Привести структуру каталогов в соответствие с `plan/04-architecture-and-api.md` и `project-structure.txt`: `src/mcp_dependency_inspector/` с подпапками `tools/`, `core/`, при необходимости `parsers/`, `scanners/`, `models/`.
- [x] Зафиксировать версии в `requirements.txt` (без `latest`); минимальный набор для v1: парсинг манифестов (tomli, ruamel.yaml), MCP SDK, httpx, pydantic; опционально — spdx-tools, license-expression.
- [x] Вынести тестовые зависимости в `requirements-dev.txt`; убедиться, что образ для production не тянет pytest и тяжёлые dev-пакеты.

### Шаг 3. Ядро: парсинг манифестов и модели данных

- [x] Реализовать `core/manifest_scanner.py`: обход директории, обнаружение `package.json`, `package-lock.json`, `pnpm-lock.yaml`, `requirements.txt`, `pyproject.toml`, `poetry.lock`; исключение директорий из конфига (например `node_modules`, `.venv`, `__pycache__`, `target`, `.git`).
- [x] Реализовать парсеры для перечисленных форматов (в `core/` или `parsers/`): выход — единая структура пакетов (name, version, ecosystem, direct/transitive).
- [x] Ввести Pydantic-модели для пакета, дерева зависимостей, лицензии, политики (в `models/` или в `core/`).

### Шаг 4. Лицензии и политика (детерминировано)

- [x] Реализовать `core/license_scanner.py`: извлечение лицензий из метаданных (package.json `license`/`licenses`, pyproject.toml, poetry.lock); классификация: permissive / copyleft / unknown / multiple по справочнику (например SPDX или встроенный маппинг).
- [x] Реализовать `core/policy_engine.py`: загрузка политики из YAML/JSON (allowed_licenses, denied_licenses, deny_unknown_license, block_critical_cve, block_high_cve); сравнение списка пакетов и лицензий с политикой; вывод списка нарушений и рекомендаций. Без сетевых вызовов.

### Шаг 5. Уязвимости: интеграция и демо-режим

- [x] Реализовать `core/vulnerability_scanner.py` (или в `scanners/`): вызов OSV API (бесплатный, без ключа) по списку пакетов; таймаут и retry; при ошибке/офлайне или при `DEMO_MODE=true` — возврат данных из стаба (фиксированный JSON с примером CVE или пустой список).
- [x] Не использовать платные API (Snyk, коммерческие базы) в базовом сценарии; опциональные ключи — только через env и не обязательны для DEMO.md.

### Шаг 6. SBOM и отчёты

- [x] Реализовать `core/sbom_builder.py`: построение графа зависимостей из выхода manifest_scanner; экспорт в SPDX 2.3 JSON и CycloneDX 1.5 JSON (минимальный набор полей).
- [x] Реализовать `core/reporter.py`: генерация отчётов в JSON и Markdown (сводки по зависимостям, лицензиям, нарушениям политики, уязвимостям).

### Шаг 7. MCP-сервер: инструменты (Tools)

- [x] Поднять MCP-сервер на Python SDK (FastMCP, stdio транспорт для v1).
- [x] Зарегистрировать инструменты с **описанием каждого параметра** (Annotated/Field):
  - `analyze_project_dependencies` (project_path, exclude_dirs) → список пакетов и сводка;
  - `scan_licenses` (project_path или packages, exclude_dirs) → список лицензий и флаги риска;

- [x] Все инструменты вызывают только детерминированные модули + опциональный HTTP (OSV); при демо-режиме — только стабы.
