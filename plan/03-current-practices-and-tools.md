# Текущие практики и инструменты: SCA, лицензии, зависимости

Обзор того, как сегодня решают задачи анализа зависимостей, уязвимостей и лицензий вне MCP — платформы, CI-инструменты и CLI.

---

## 1. Классические SCA-платформы

Software Composition Analysis (SCA) — сканирование открытого кода на уязвимости, лицензии и соответствие политикам.

### Общие возможности

- **Дерево зависимостей:** полный граф direct + transitive по манифестам (package.json, requirements.txt, pom.xml и т.д.) или по бинарникам.
- **Уязвимости:** сверка с CVE/NVD, OSV, GitHub Advisory, собственные базы; уровни серьёзности, рекомендации по обновлению.
- **Лицензии:** определение лицензий пакетов, whitelist/blacklist, политики компании, отчёты для юристов.
- **SBOM:** генерация Software Bill of Materials (SPDX, CycloneDX) для регуляторов и заказчиков.
- **Интеграции:** CI/CD (GitHub Actions, GitLab CI, Jenkins), issue-трекеры, веб-дашборды.

### Типичный UX

- Скан при каждом пуше/PR; блокировка мержа при нарушении политики.
- Боты (Dependabot, Renovate, встроенные в платформу) создают PR с обновлениями.
- Веб-консоль: дашборды, отчёты, настройка политик, история сканов.

### Основные платформы

| Платформа | Особенности | Лицензии | Мульти-язык |
|-----------|-------------|----------|--------------|
| **Snyk** | SCA + SAST + контейнеры; удобный UX, интеграции с GitHub/GitLab. | Да, политики | Да |
| **Black Duck (Synopsys)** | Крупная корпоративная SCA, бинарный анализ, полноценный SBOM. | Да, детальный анализ | Да |
| **FOSSA** | Сильный акцент на license compliance (99.8% accuracy по заявлениям), SBOM, политики. | Основной фокус | Да |
| **Semgrep Supply Chain** | Сканирование зависимостей + правила Semgrep; license compliance в документации. | Политики, allow/deny/comment | Да |
| **Anchore / Grant** | Контейнеры и образы, SBOM, лицензии; open-source компоненты. | Да | Да (через образы) |
| **JFrog Xray** | Интеграция с Artifactory; сканирование артефактов и зависимостей. | Да | Да |

**Ограничения для сценария «ИИ в диалоге»:** результаты живут в веб-консоли и CI; нет единого MCP-интерфейса для запросов вида «покажи все GPL в продакшен-пути»
или «сформируй отчёт для юриста» прямо из чата.

## 2. DevOps-инструменты и CI-плагины

Фокус на автоматическом обновлении версий и блокировке небезопасных зависимостей в пайплайне.

| Инструмент | Назначение | Уязвимости | Лицензии | Примечание |
|------------|------------|------------|----------|------------|
| **Dependabot** | Авто-PR с обновлениями зависимостей (GitHub). | Да (GitHub Advisory) | Ограниченно | Встроен в GitHub; не правит код при breaking changes. |
| **Renovate** | Апдейты зависимостей, группировка PR, кастомные правила. | Да (интеграции) | Опционально | Гибкая настройка; сам код не меняет. |
| **GitHub Advanced Security / Dependabot** | Сканирование репо, алерты, Dependabot alerts. | Да | Нет в базовом сценарии | Часть GitHub. |
| **Trivy** | Сканирование образов, файловых систем, IaC; открытый, CLI и CI. | Да (OSV и др.) | Да (лицензии в образах) | Популярен в контейнерном стеке. |
| **OWASP Dependency-Check** | Сканирование зависимостей по NVD и другим базам. | Да | Нет | Бесплатный, часто в CI. |

**Практика:** скан на каждом PR; при критических CVE — fail или мандат на апдейт. Лицензии чаще настраивают в полноценных SCA-платформах (Snyk, FOSSA и т.д.),
а не в Dependabot/Renovate.

**Ограничения:** нет интерактивного диалога; разработчик смотрит алерты в UI или в логах CI. ИИ не имеет прямого доступа к структурированным результатам
без отдельного API/MCP.

## 3. CLI и библиотечные утилиты

Запуск вручную или из скриптов; результаты — текст/JSON в stdout.

### Уязвимости

| Экосистема | Инструменты |
|------------|-------------|
| **JavaScript/TypeScript** | `npm audit`, `yarn audit`, `pnpm audit` — встроенные; вывод по проекту. |
| **Python** | `pip-audit` (OSV), `safety` — по requirements; вывод CVE и рекомендаций. |
| **JVM** | Maven/Gradle плагины (OWASP Dependency-Check, Snyk plugin и др.) — проверка по CVE. |
| **Универсальные** | `trivy` — образы, fs, зависимости по манифестам. |

### Лицензии

| Экосистема | Инструменты |
|------------|-------------|
| **JavaScript** | `license-checker`, `npm-license-crawler` — обход дерева, вывод лицензий. |
| **Python** | `pip-licenses` — список лицензий установленных пакетов. |
| **Универсальные** | `scancode-toolkit` — глубокий скан кода и лицензий; `licensee` — определение лицензии репозитория. |

**Недостатки:** разрозненность по языкам; вывод нужно самому интерпретировать и сводить в отчёт; для юристов и compliance сырой вывод неудобен.
Нет единого «одной командой по всему проекту» с учётом нескольких экосистем.

## 4. Типичные потоки работы

### Разработчик перед PR

1. Запускает локально `npm audit` / `pip-audit` / плагин в сборке.
2. Смотрит список CVE; решает, что править в этом PR.
3. Лицензии проверяют реже; обычно при онбординге или по запросу юристов.

### CI/CD

1. На каждом пуше/PR — запуск SCA (Snyk, Trivy, OWASP Dependency-Check и т.д.).
2. При нарушении политики — fail или предупреждение; иногда авто-PR от Dependabot/Renovate.
3. Результаты в веб-интерфейсе платформы или в артефактах CI.

### Аудит для безопасников / compliance

1. Разовый или периодический полный скан репо/артефактов.
2. Экспорт SBOM и отчётов по лицензиям.
3. Сверка с корпоративной политикой вручную или через правила в SCA.

**Проблема:** между «сырыми» результатами и ответами на вопросы типа «все ли GPL в dev-зависимостях?» или «что нужно исправить в первую очередь?» — разрыв.
ИИ с MCP может заполнить этот разрыв, если получает структурированные данные (дерево зависимостей, лицензии, CVE, политики) и формирует ответы в диалоге.

---

## 5. Недостатки текущих подходов для интеграции с ИИ

- **Нет единого API под «вопрос-ответ»:** SCA и CI выдают отчёты и алерты, но не отвечают на произвольные запросы в контексте чата.
- **Разделение по экосистемам:** один инструмент — один язык/менеджер; кросс-языковой сводный отчёт собирают вручную или в тяжёлых платформах.
- **Лицензии вторичны во многих инструментах:** фокус на CVE; политики лицензий и отчёты для юристов есть в основном в платных SCA.
- **Нет стандартного протокола для агентов:** MCP даёт единый способ подключить «инспекцию зависимостей и лицензий» к любому MCP-клиенту (Cursor, Claude, VS Code и т.д.)
  и получать структурированные ответы в диалоге.

---

## 6. Вывод

Текущие практики покрывают:

- **Сканирование и алерты** — SCA-платформы, Trivy, Dependabot, Renovate, npm/pip audit.
- **Лицензии и политики** — в основном в платформах (FOSSA, Snyk, Black Duck и др.) и частично в CLI.
- **SBOM и отчёты** — генерация есть; потребление часто вручную или через отдельные системы.

**Ниша MCP «Инспектор зависимостей и лицензий»:** единая точка входа для ИИ-клиентов — один сервер даёт анализ по нескольким экосистемам, лицензии с политиками, SBOM и отчёты,
и всё это доступно в виде инструментов и ресурсов в диалоге, без обязательной привязки к конкретному SaaS.
