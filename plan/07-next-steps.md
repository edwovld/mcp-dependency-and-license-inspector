# 07. Следующие шаги развития проекта

## 1. Цели

- Сфокусировать усилия на «тяжёлых» и творческих задачах: документация, демо‑сценарии, материалы для защиты.
- Получить дополнительные материалы вокруг MCP‑сервера без изменения его детерминированного ядра.
- Подготовить документацию и материалы для демонстрации и защиты проекта.

---

## 2. Порядок выполнения

Шаги выполняются **в указанном порядке**: каждый следующий опирается на контекст и артефакты предыдущих. После каждого шага результат сохраняется в репозиторий (новые файлы или правки в `plan/`, `docs/`, `presentation/`), чтобы следующий шаг мог на них опираться.

| №   | Задача                          | Зачем                                                        | Ожидаемый результат                          |
| --- | ------------------------------- | ------------------------------------------------------------ | -------------------------------------------- |
| 1   | Резюме продукта                 | Единая формулировка «что это и для кого» для всех материалов | Краткие тексты (RU/EN) в `docs/` или в плане |
| 2   | Варианты README                 | Разные аудитории — разный уровень детализации                | Черновики/альтернативы README                |
| 3   | Расширение DEMO и кейсы         | Готовые сценарии для показа и тестирования                   | Обновлённый DEMO.md + описание кейсов        |
| 4   | FAQ                             | Ответы на типичные вопросы до защиты                         | Файл FAQ (например, `docs/FAQ.md`)           |
| 5   | Презентация и скрипт докладчика | Слайды + подсказки для выступления                           | Правки в Slidev + скрипт докладчика          |
| 6   | Вопросы от жюри и ответы        | Подготовка к защите                                          | Список Q&A в `docs/` или в плане             |
| 7   | Roadmap и позиционирование      | Стратегия развития и отличия от конкурентов                  | Roadmap v2/v3 + краткий конкурентный обзор   |

---

## 3. Задачи и формулировки (по порядку)

Ниже — формулировки для пошагового выполнения. Можно передавать их агенту или использовать как чек‑лист при ручной работе.

### Задача 1 — Резюме продукта (RU + EN)

```
Контекст: репозиторий с MCP‑сервером «Инспектор зависимостей и лицензий». Прочитай plan/01–07, README.md и .cursorrules.

Задача: напиши два коротких резюме продукта (по 2–3 абзаца каждое):
1. На русском — для технического руководителя или жюри (что это, для кого, главная ценность, ограничения).
2. На английском — для GitHub / внешней аудитории (то же по смыслу, формулировки под англоязычный контекст).

Требования: без воды, без маркетинговых клише. Упомяни: детерминированное ядро, MCP tools (analyze, scan, check_policy, generate_sbom, suggest), DEMO без платных API, Docker‑контракт. Результат сохрани в docs/product-summary.md (или предложи путь и создай файл).
```

### Задача 2 — Варианты README для разных аудиторий

```
Контекст: в репозитории есть README.md и docs/product-summary.md (или резюме из плана 07). Архитектура и API описаны в plan/05-architecture-and-api.md.

Задача: подготовь два варианта текста README (можно как отдельные файлы или как секции в одном файле):
1. Краткий питч (1–2 абзаца) для руководства: что за продукт, зачем, один пример использования.
2. Подробное описание для разработчиков и AppSec: установка, запуск (serve/smoke), основные tools, примеры вызовов, ограничения по ресурсам и безопасности из .cursorrules.

Сохрани черновики в docs/README-pitch.md и docs/README-developers.md (или обнови основной README с явными секциями «Кратко» и «Для разработчиков»). Не меняй логику и контракты из plan/05.
```

### Задача 3 — Расширение DEMO.md и новые кейсы

```
Контекст: в репозитории есть DEMO.md с базовым сценарием. Tools описаны в plan/05-architecture-and-api.md. Требование: все сценарии выполнимы без платных сервисов и API‑ключей (DEMO_MODE или стабы).

Задача:
1. Дополни DEMO.md тремя новыми сценариями (пошагово, с ожидаемыми результатами):
   - монорепозиторий с несколькими сервисами (несколько манифестов),
   - проект с жёсткой лицензионной политикой (проверка check_policy_compliance),
   - проект с акцентом на историю миграции зависимостей (если поддерживается — иначе опиши ограничение).
2. Добавь раздел «Экстремальные кейсы»: очень большой requirements.txt, смешанные экосистемы (Python + Node), конфликтующие политики лицензий — что ожидать от сервера и как интерпретировать вывод.

Сохрани изменения в DEMO.md. Если нужны фикстуры или примеры манифестов — предложи пути и создай минимальные примеры в tests/fixtures/ или в docs/.
```

### Задача 4 — FAQ по MCP‑серверу

```
Контекст: plan/01–07, README, DEMO.md, .cursorrules. Продукт — MCP‑сервер для анализа зависимостей, лицензий, уязвимостей и проверки политики.

Задача: составь FAQ из 10–15 вопросов и кратких ответов. Включи:
- что такое MCP и зачем этот сервер,
- чем отличается от Dependabot / Snyk / других SCA,
- как запустить локально и в Docker,
- что такое DEMO_MODE и когда использовать стабы,
- как задать политику лицензий,
- ограничения (детерминизм, размер образа, ресурсы),
- как расширить под новые экосистемы (roadmap).

Сохрани в docs/FAQ.md. Формат: вопрос (заголовок) — ответ (1–3 абзаца, без лишней воды).
```

## 4. Альтернатива: одна объединённая сессия

Вместо пошагового выполнения можно провести одну стратегическую сессию по всем темам. Ниже — объединённая формулировка; результаты по-прежнему сохранять по разделам (docs/, plan/) для последующего использования.

```
Ты — ведущий архитектор и продуктовый менеджер MCP‑сервера «Инспектор зависимостей и лицензий».

У меня есть детерминированный MCP‑сервер для анализа зависимостей, лицензий, уязвимостей и проверки политики лицензий. Ядро на Python 3.11, MCP SDK (stdio), Pydantic, tomli, ruamel.yaml, httpx, OSV (или стабы в DEMO_MODE). В репозитории: планы plan/01–07, рабочий сервер с tools (analyze_project_dependencies, scan_vulnerabilities, scan_licenses, check_policy_compliance, generate_sbom, suggest_dependency_replacements), DEMO‑сценарий, Docker‑контракт (serve/smoke), тесты и презентация на Slidev.

Задача: помочь спланировать следующие 2–3 итерации и подготовить материалы для демонстрации и защиты.

Сделай по порядку:

1. **Резюме продукта** — 2–3 абзаца на русском и на английском для техруководства / GitHub.

3. **Новые DEMO‑кейсы** — 3–4 сценария с шагами и ожидаемыми результатами (монорепо, жёсткая политика лицензий, экстремальные кейсы).
4. **Вопросы от жюри/аудитории** — 15–20 вопросов с краткими содержательными ответами (архитектура, ограничения, безопасность, сравнение с SCA).
5. **Улучшения презентации** — какие слайды добавить/объединить, какие метрики или диаграммы показать; черновик тезисов для докладчика.

Формат: разделы 1–5 с нумерацией, списки и короткие формулировки без воды. Результаты предлагай сохранять в docs/ и plan/ с конкретными именами файлов.
```

## 5. Сводка тем

Перечень тем для справки; детали и порядок — в разделах 2–4.

- **Документация:** альтернативные README (питч / для разработчиков), описание на английском, FAQ 10–15 вопросов.
- **DEMO:** новые сценарии (монорепо, политика лицензий, история миграций), экстремальные кейсы (большой requirements.txt, Python+Node, конфликтующие политики).
- **Презентация:** правки слайдов, скрипт докладчика, список вопросов от жюри с ответами.
- **Roadmap:** v2/v3, коммерциализация, метрики для бизнеса.
- **Позиционирование:** обзор SCA‑инструментов, сравнение с нашим MCP‑сервером, сильные и слабые стороны.

После выполнения шагов 1–7 (или объединённой сессии) обнови этот план: отметь выполненные пункты и при необходимости добавь ссылки на созданные файлы в `docs/` и `plan/`.
