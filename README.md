# Dependency & License Inspector MCP

Детерминированный MCP-сервер для инспекции зависимостей и лицензий: сканирование манифестов (npm, PyPI, Poetry), проверка лицензий и политик, сканирование уязвимостей (OSV или стабы в демо-режиме), генерация SBOM (SPDX/CycloneDX).

---

## Что делает MCP-сервер

- **Единая точка входа** — все запросы по зависимостям и лицензиям через один MCP-сервер из любого MCP-клиента (Cursor, Claude Code, VS Code, MCP Inspector).
- **Кросс-языковой анализ** — один проект, один граф зависимостей: JavaScript/TypeScript (npm, yarn, pnpm), Python (pip, poetry).
- **Скан уязвимостей** — интеграция с OSV API; агрегированный список CVE с severity и рекомендуемыми версиями; в демо-режиме — стабы без сети.
- **Скан лицензий** — сбор лицензий из метаданных пакетов; флаги риска (copyleft, unknown, multiple).
- **Проверка политик** — движок политик (YAML/JSON): допустимые/запрещённые лицензии, блокировка по CVE; отчёт о нарушениях и рекомендациях.
- **SBOM и отчёты** — генерация SBOM (SPDX 2.3, CycloneDX 1.5) и сводок в Markdown/JSON.

Без LLM в ядре: результаты определяются только входными данными (манифесты, конфиг политики).

---

## Какую боль решает

- **Ручной аудит по разным инструментам** — вместо часов запуска npm audit, pip-audit и просмотра консолей: один запрос в MCP — полная картина по зависимостям, уязвимостям и лицензиям.
- **Риски CVE и лицензий** — раннее обнаружение критичных CVE и недопустимых лицензий до мержа или релиза; приоритизированные списки и рекомендации по заменам.
- **Разрозненные экосистемы** — один протокол (MCP) и один набор инструментов для npm и Python в одном проекте.
- **Готовность к регуляторам** — автоматическая генерация SBOM и отчётов для прозрачности цепочки поставок.

## Какие инструменты реализованы

Реализовано **6 инструментов (tools)**:

| Инструмент                        | Назначение                                                        |
| --------------------------------- | ----------------------------------------------------------------- |
| `analyze_project_dependencies`    | Граф зависимостей (direct + transitive) по манифестам npm/Python. |
| `scan_vulnerabilities`            | Уязвимости через OSV API или стабы в DEMO_MODE.                   |
| `scan_licenses`                   | Лицензии и флаги риска (copyleft, unknown, multiple).             |
| `check_policy_compliance`         | Проверка по политике (YAML/JSON): нарушения и рекомендации.       |
| `generate_sbom`                   | SBOM в формате SPDX 2.3 или CycloneDX 1.5 (JSON).                 |
| `suggest_dependency_replacements` | Рекомендации по замене проблемных пакетов (CVE/лицензия).         |

Каждый инструмент выполняет одну понятную операцию, имеет чёткую схему входных данных (типы и описания параметров), возвращает предсказуемый JSON без скрытых побочных эффектов (кроме обновления сессионного состояния для Resources — см. ниже).

**Описание инструмента — промпт для модели.** При подключении сервер отдаёт имя, описание и схему параметров каждого tool; модель по ним решает, какой инструмент вызвать. Расплывчатое описание («работает с данными») бесполезно; конкретное («возвращает список CVE с severity и рекомендуемыми версиями; в DEMO_MODE — стаб») позволяет модели корректно выбирать инструмент.

## Транспорт и протокол MCP

Сервер поддерживает оба варианта транспорта, принятых в MCP:

| Транспорт           | Назначение                                                                                                                                                                  |
| ------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **stdio**           | Локальный запуск: клиент (Cursor, Claude Code, MCP Inspector) запускает процесс и общается через stdin/stdout. Просто, быстро, без сети.                                    |
| **Streamable HTTP** | Удалённый доступ: HTTP-запросы и стриминг через Server-Sent Events. Подходит для облачных деплоев и Docker; endpoint — `http://localhost:8000/mcp` при пробросе порта 8000. |

Под капотом используется **JSON-RPC 2.0**: каждый вызов инструмента — пара запрос/ответ. Всё типизировано через **JSON Schema** — аргументы инструментов описываются схемой, которую MCP SDK формирует из сигнатур и аннотаций (`Annotated[тип, "описание"]`), поэтому модель знает, какие аргументы передать.

## Resources (ресурсы)

Данные, к которым агент получает доступ через MCP Resources — только чтение, без побочных эффектов.

| URI                       | Описание                                                                                                       | Ограничения                                                              |
| ------------------------- | -------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------ |
| `report://latest`         | Последний отчёт о соответствии политике (результат последнего вызова `check_policy_compliance` в этой сессии). | Только сессия; после перезапуска сервера пусто. JSON.                    |
| `sbom://latest/spdx`      | Последний сгенерированный SBOM в формате SPDX 2.3 JSON.                                                        | Заполняется после вызова `generate_sbom(..., format='spdx')`. JSON.      |
| `sbom://latest/cyclonedx` | Последний сгенерированный SBOM в формате CycloneDX 1.5 JSON.                                                   | Заполняется после вызова `generate_sbom(..., format='cyclonedx')`. JSON. |

**Разрешённые источники:** только состояние последнего скана/отчёта в памяти процесса. Объём и тип данных: JSON; объём ограничен размером графа зависимостей и SBOM (рекомендуемые лимиты: CPU 2.0, RAM 2048 MB — см. Docker).

## Prompts (промпты)

Шаблоны запросов для стандартизации операций: агент получает инструкцию, какую последовательность инструментов вызвать и в каком формате представить результат.

| Имя                            | Назначение                                                                                                                                                                                                                     |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `audit_dependencies_and_risks` | Аудит зависимостей и рисков: последовательность вызовов (analyze → scan_vulnerabilities → scan_licenses → check_policy_compliance), сводка по критичным/высоким рискам и рекомендации через `suggest_dependency_replacements`. |
| `license_report_for_legal`     | Отчёт по лицензиям для юриста: вызовы `scan_licenses` и `check_policy_compliance`, таблица по типам лицензий и рекомендуемые действия по copyleft/unknown.                                                                     |

Промпты не вызывают LLM внутри сервера — они возвращают текст-инструкцию для клиента (агента), который сам выполняет вызовы tools и интерпретирует результат.

## Быстрый старт

Демо запускается только через **Docker** (без установки Python и зависимостей на хост). Для визуальной проверки нужен **MCP Inspector** (запуск через `npx`, нужен Node.js). Выполняйте шаги по порядку из корня репозитория.

### Шаг 1. Собрать образ

```bash
cd /path/to/mcp-codenrock
docker build -t dependency-inspector-mcp .
```

(Подставьте свой путь к репозиторию вместо `/path/to/mcp-codenrock`.)

### Шаг 2. Проверить готовность (по желанию)

```bash
docker run --rm dependency-inspector-mcp smoke
```

Успех: в конце вывода «smoke: ok — /health returned 200 and status ok», код выхода 0.

### Шаг 3. Запустить MCP-сервер

```bash
docker run --rm -p 8000:8000 --memory=2048m --cpus=2 -e DEMO_MODE=true dependency-inspector-mcp serve
```

Терминал оставьте открытым. Сервер будет слушать порт 8000; endpoint для клиента: `http://localhost:8000/mcp`. Остановка: Ctrl+C.

### Шаг 4. Подключить MCP Inspector к серверу

1. Установите **Node.js** (если ещё нет): https://nodejs.org/
2. В **новом терминале** (сервер из шага 3 продолжает работать) выполните:
   ```bash
   npx -y @modelcontextprotocol/inspector
   ```
3. В браузере откроется MCP Inspector. В панели подключения (справа):
   - **Transport** выберите **Streamable HTTP** (или «Streamable HTTP» / HTTP с SSE).
   - В поле **URL** введите: `http://localhost:8000/mcp`
   - Нажмите **Connect** (или «Подключить»).
4. После подключения в списке появятся инструменты сервера (например `check_policy_compliance`, `analyze_project_dependencies` и др.).

**Важно:** в браузере по прямой ссылке `http://localhost:8000/mcp` будет ошибка 406 — так и задумано. Подключаться к серверу нужно только через MCP Inspector (или другой MCP-клиент), а не через адресную строку браузера.

### Шаг 5. Проверить демо-сценарий

В MCP Inspector вызовите два инструмента по очереди, как описано в **[DEMO.md](DEMO.md)** (проверка политики и рекомендации по заменам). Так вы увидите работу MCP на встроенном демо-проекте.

## Как использовать

Краткое описание каждого инструмента, входные параметры и ожидаемый результат. Тестовый проект для проверки: **`demo_project/`** в корне репозитория (см. раздел «Ограничения / допущения»).

### 1. `analyze_project_dependencies`

**Описание:** Сканирует корень проекта и возвращает полный граф зависимостей (прямые и транзитивные) по всем обнаруженным экосистемам (npm, PyPI).

| Параметр       | Тип          | Обязательный | Описание                                                                                                                 |
| -------------- | ------------ | ------------ | ------------------------------------------------------------------------------------------------------------------------ |
| `project_path` | string       | да           | Путь к корневой директории проекта.                                                                                      |
| `exclude_dirs` | list[string] | нет          | Имена директорий, исключаемых из сканирования (по умолчанию из конфига: node_modules, .venv, **pycache**, target, .git). |

**Результат:** JSON с полями: `summary` (Markdown-сводка), `total_packages`, `direct_count`, `transitive_count`, `ecosystem_counts`, `packages` (массив объектов с name, version, ecosystem, direct и т.д.).

---

### 2. `scan_vulnerabilities`

**Описание:** Сканирует зависимости проекта на известные уязвимости (OSV). В режиме `DEMO_MODE=true` или при недоступности API возвращает стаб-данные.

| Параметр       | Тип          | Обязательный | Описание                                 |
| -------------- | ------------ | ------------ | ---------------------------------------- |
| `project_path` | string       | да           | Путь к корневой директории проекта.      |
| `exclude_dirs` | list[string] | нет          | Директории, исключаемые из сканирования. |

**Результат:** JSON: `summary`, `counts_by_severity`, `packages_scanned`, `vulnerabilities` (массив с пакетом, версией, CVE, severity, рекомендуемая версия), `demo_mode` (bool).

---

### 3. `scan_licenses`

**Описание:** Список лицензий всех зависимостей с флагами риска (copyleft, unknown, multiple). Можно передать путь к проекту или явный список пакетов.

| Параметр       | Тип          | Обязательный | Описание                                                                          |
| -------------- | ------------ | ------------ | --------------------------------------------------------------------------------- |
| `project_path` | string       | нет\*        | Путь к корню проекта. \*Обязателен один из: project_path или packages.            |
| `packages`     | list[object] | нет\*        | Список объектов вида {name, version?, ecosystem?} для проверки без скана с диска. |
| `exclude_dirs` | list[string] | нет          | Директории, исключаемые при сканировании по project_path.                         |

**Результат:** JSON: `summary`, `total_packages`, `licenses` (массив с полями package, version, ecosystem, license, spdx_id, category, is_copyleft, is_unknown, is_multiple).

---

### 4. `check_policy_compliance`

**Описание:** Проверяет проект на соответствие политике лицензий и CVE (allowed/denied licenses, block_critical_cve, block_high_cve, deny_unknown_license).

| Параметр       | Тип               | Обязательный | Описание                                                                                                                                                                                                  |
| -------------- | ----------------- | ------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `project_path` | string            | да           | Путь к корневой директории проекта.                                                                                                                                                                       |
| `policy`       | object или string | нет          | Политика: объект с ключами allowed_licenses, denied_licenses, deny_unknown_license, block_critical_cve, block_high_cve либо путь к YAML/JSON файлу. Если не передан — используется политика по умолчанию. |
| `exclude_dirs` | list[string]      | нет          | Директории, исключаемые из сканирования.                                                                                                                                                                  |

**Результат:** JSON: `summary`, `compliant` (bool), `packages_checked`, `violations_count`, `violations` (массив нарушений с типом и деталями), `demo_mode`.

---

### 5. `generate_sbom`

**Описание:** Генерирует Software Bill of Materials в выбранном формате.

| Параметр       | Тип          | Обязательный | Описание                                                         |
| -------------- | ------------ | ------------ | ---------------------------------------------------------------- |
| `project_path` | string       | да           | Путь к корневой директории проекта.                              |
| `format`       | string       | да           | `"spdx"` (SPDX 2.3 JSON) или `"cyclonedx"` (CycloneDX 1.5 JSON). |
| `exclude_dirs` | list[string] | нет          | Директории, исключаемые из сканирования.                         |

**Результат:** JSON: `format`, `package_count`, `sbom` (объект SBOM в выбранном формате).

---

### 6. `suggest_dependency_replacements`

**Описание:** Предлагает альтернативы для проблемных зависимостей (CVE или недопустимая лицензия). В DEMO_MODE или при ошибке запроса — возвращаются стаб-рекомендации.

| Параметр   | Тип               | Обязательный | Описание                                                                                             |
| ---------- | ----------------- | ------------ | ---------------------------------------------------------------------------------------------------- |
| `packages` | list[object]      | да           | Список пакетов для замены: [{name, version?, ecosystem?, reason}], где reason — "cve" или "license". |
| `policy`   | object или string | нет          | Опциональная политика (объект или путь к файлу) для фильтрации предлагаемых замен по лицензии.       |

**Результат:** JSON с предложениями по каждому пакету: альтернативы с name, version, license, reason (краткое обоснование); в демо-режиме — стаб-данные.

## Ограничения / допущения

### Что поддерживается

- **Манифесты:** `package.json`, `package-lock.json`, `pnpm-lock.yaml`, `requirements.txt`, `pyproject.toml`, `poetry.lock`.
- **Экосистемы:** npm (Node.js), PyPI (Python).
- **Уязвимости:** OSV API (бесплатно, без ключа); при `DEMO_MODE=true` или офлайне — стабы.
- **Лицензии:** из метаданных манифестов; классификация permissive / copyleft / unknown / multiple.
- **Политика:** YAML или JSON с правилами по лицензиям и CVE.
- **SBOM:** SPDX 2.3 JSON, CycloneDX 1.5 JSON.
- **Транспорты:** stdio (локальный процесс), Streamable HTTP на порту 8000 (Docker).

### Что не поддерживается

- Вызовы LLM в ядре (сервер детерминированный).
- Maven/Gradle (только npm и Python в текущей версии).
- Платные API (Snyk и т.п.) в базовом сценарии.
- Хранение секретов в репозитории или в образе.

### На каком тестовом проекте проверялось

- **`demo_project/`** в корне репозитория: тестовые манифесты npm и Python, пакет с лицензией GPL (`pkg_with_gpl`), политика `policy_strict.yaml`. Все 6 инструментов проверяются на этом пути без платных сервисов и API-ключей при `DEMO_MODE=true`. Подробный сценарий — в [DEMO.md](DEMO.md).

## (Опционально) Расширенный режим

### Внешние интеграции

- **OSV API** — используется по умолчанию для сканирования уязвимостей. Ключ не требуется. Базовый URL задаётся переменной `OSV_API_BASE` (по умолчанию `https://api.osv.dev`). При недоступности сети или при `DEMO_MODE=true` используются стабы.
- **Дополнительные источники** (например Libraries.io) — опционально через переменную окружения для API-ключа; в базовом и демо-сценарии не обязательны. Имя переменной указывается в конфиге (например `LIBRARIES_IO_API_KEY`); значение не публикуется в репозитории.

### Как включить

1. Скопировать пример конфигурации окружения:
   ```bash
   cp .env.example .env
   ```
2. В `.env` выставить нужные переменные (см. комментарии в `.env.example`). Для работы с реальным OSV без демо-режима:
   ```bash
   DEMO_MODE=false
   ```
3. Секреты (API-ключи) задавать только через переменные окружения или защищённый конфиг, не коммитить в репозиторий. В `.env.example` указаны только имена переменных и пустые/примерные значения.

### Что требуется (без публикации секретов)

- Сеть для доступа к OSV API при `DEMO_MODE=false`.
- Опционально: ключи к дополнительным сервисам в переменных окружения, как описано в `.env.example`.

### Ограничения доступа MCP (безопасность)

Настраиваются переменными окружения (значения по умолчанию — в таблице). Подробнее — в [plan/04-architecture-and-api.md](plan/04-architecture-and-api.md).

| Параметр                   | Описание                                                                                 | По умолчанию     |
| -------------------------- | ---------------------------------------------------------------------------------------- | ---------------- |
| `ALLOWED_BASE_PATH`        | Если задан — все `project_path` в инструментах должны находиться внутри этой директории. | Не задан.        |
| `MAX_INPUT_PATH_LENGTH`    | Максимальная длина пути (символов).                                                      | 4096             |
| `MAX_EXCLUDE_DIRS`         | Максимальное число элементов в `exclude_dirs`.                                           | 100              |
| `MAX_POLICY_PAYLOAD_BYTES` | Максимальный размер политики (байт): путь или JSON.                                      | 524288 (512 KiB) |
| `TOOL_TIMEOUT_SECONDS`     | Таймаут выполнения одного инструмента (с). 0 = отключено.                                | 120              |

Сервер **не выполняет произвольные shell-команды**; все операции — чтение файлов и HTTP-запросы (например, OSV API).

## Тесты

Требуется Python 3.11+. Установите пакет и dev-зависимости:

```bash
pip install -e .
pip install -r requirements-dev.txt
```

Запуск тестов (без LLM и без платных API; при DEMO_MODE используются стабы):

```bash
export DEMO_MODE=true
python -m pytest tests/ -v --tb=short
```

Если пакет не установлен в venv:

```bash
export PYTHONPATH=src:$PYTHONPATH
```

(и зависимости из `requirements.txt` должны быть установлены).
